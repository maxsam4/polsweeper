Architecture Approaches

How should the factory, proxies, and implementation contract relate?

A

Factory + Shared Implementation (Recommended)

One VirtualAccountCreator factory deploys EIP-1167 clones via CREATE2. All clones delegate to a single VirtualAccountImpl contract. The impl stores nothing — it reads the master address from the clone's immutable args (baked into the proxy bytecode).

┌─────────────────────────────┐
│   VirtualAccountCreator     │
│   (Factory)                 │
│                             │
│  createAccounts(master, n)  │
│    → CREATE2 + EIP-1167     │
│    → deterministic addrs    │
│                             │
│  getAddress(master, idx)    │
│    → precompute address     │
└──────────┬──────────────────┘
           │ deploys clones
           ▼
┌──────────────────┐  ┌──────────────────┐
│  Clone 0         │  │  Clone 1         │
│  (45 bytes)      │  │  (45 bytes)      │
│  master=0xABC..  │  │  master=0xABC..  │
│  delegatecall ──►│  │  delegatecall ──►│
└──────────┬───────┘  └──────────┬───────┘
           │                     │
           ▼                     ▼
     ┌─────────────────────────────┐
     │   VirtualAccountImpl        │
     │   (Shared singleton)        │
     │                             │
     │  sweepPOL()                 │
     │    → send balance to master │
     │                             │
     │  sweepERC20(token)          │
     │    → transfer to master     │
     └─────────────────────────────┘

Pros

  • Cheapest deployment (~$0.001 per clone)
  • Master baked into bytecode = no storage reads
  • Deterministic addresses before deployment
  • Single impl to audit

Cons

  • Master cannot be changed (but we chose this)
  • Impl upgrade requires redeploying clones
B

Factory + Storage-based Impl

Same factory pattern, but each clone stores its master address in contract storage (set via initialize()). The impl reads master from storage on each sweep.

┌─────────────────────────────┐
│   VirtualAccountCreator     │
│   (Factory)                 │
│                             │
│  createAccounts(master, n)  │
│    → CREATE2 + EIP-1167     │
│    → clone.initialize(master)│
└──────────┬──────────────────┘
           │ deploys + initializes
           ▼
┌──────────────────┐  ┌──────────────────┐
│  Clone 0         │  │  Clone 1         │
│  storage:        │  │  storage:        │
│    master=0xABC  │  │    master=0xABC  │
│  delegatecall ──►│  │  delegatecall ──►│
└──────────┬───────┘  └──────────┬───────┘
           │                     │
           ▼                     ▼
     ┌─────────────────────────────┐
     │   VirtualAccountImpl        │
     │   (reads master from slot)  │
     └─────────────────────────────┘

Pros

  • More familiar pattern (OpenZeppelin Clones)
  • Could support updatable master if needed later

Cons

  • Higher deploy cost (initialize tx per clone)
  • SLOAD on every sweep = higher gas
  • Storage slot collision risks
  • Needs initializer guard
C

ClonesWithImmutableArgs (CWIA)

Uses CWIA library to append immutable data (master address) after the clone bytecode. The impl reads master from calldata/code, not storage.

┌─────────────────────────────┐
│   VirtualAccountCreator     │
│   (Factory using CWIA)      │
│                             │
│  createAccounts(master, n)  │
│    → CWIA.clone(impl,       │
│        abi.encode(master),  │
│        salt)                │
└──────────┬──────────────────┘
           │
           ▼
┌──────────────────────────────┐
│  Clone (~65 bytes)           │
│  bytecode tail = master addr │
│  delegatecall to impl ──────►│
└──────────────────────────────┘
           │
           ▼
     ┌─────────────────────────────┐
     │   VirtualAccountImpl        │
     │   _getArgAddress(0) = master│
     └─────────────────────────────┘

Pros

  • No storage needed (reads from bytecode)
  • Gas-efficient reads
  • Clean pattern for immutable per-clone data

Cons

  • Extra dependency (CWIA library)
  • Slightly larger clone bytecode (~65 vs 45 bytes)
  • Less widely adopted than standard 1167
